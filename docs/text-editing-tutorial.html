PDFDancer Java SDK · Text editing

Edit PDF text with Java

A text-editing reference for the PDFDancer Java SDK, with minimal examples for selection, replacement, insertion, deletion, styling, verification, and saving.

Scope

Prerequisites are Java 17+, a configured SDK, and an active PDFDancer session named pdf. Only text-editing APIs are covered.

Each operation example is independent. Run it against a fresh session to reproduce the described result without changes from earlier examples.

01 · Mental model

How a text-editing request works

Text edits operate on the PDF in the active session. A request identifies text or an insertion position, specifies one operation, and returns a TextEditResponse. Applied changes remain in the session until the PDF is saved or downloaded.

1

Identify the target

Use literal text, a regular expression, a style-run filter, an anchor caret, or page coordinates. The available target types depend on the operation.

2

Specify the operation and layout

Replace, insert, delete, or style the target. Choose source-anchored placement or a reflow mode based on the required output layout.

3

Verify and save

Inspect TextEditResponse before saving. A request can find matches but apply fewer changes, return warnings, or return errors.

SESSION PREREQUISITE

Create or obtain the active session

PDFDancer pdf = PDFDancer.createSession("input.pdf");

Prerequisite: the application has already configured the dependency, authentication, and required imports.

Page numbers are one-based

pdf.page(1) means the first page. Calling an operation through pdf.text() makes it document-scoped; calling it through pdf.page(n).text() scopes it to that page.

02 · Targeting text

Select the text to edit

Literal and regular-expression selectors are available for replacement, insertion, deletion, and styling. Style requests also support filters on existing text runs. Page scope can narrow any selector to specific pages.

Selector fragments

The snippets below are incomplete builder chains. They contain selector and scope configuration only; mutation configuration, build(), and execution are omitted.

LITERAL SELECTOR

Match literal text

TextReplaceRequest.builder()
        .literal("Draft")
        .caseSensitive(false)
        .wholeWords(true)
        .maxMatches(1);

Selection: the first whole-word occurrence of “Draft”, regardless of letter case. A substring such as “Drafting” is excluded.

REGULAR-EXPRESSION SELECTOR

Match a regular expression

TextDeleteRequest.builder()
        .regex("INV-\\d{6}");

Selection: text such as “INV-240731”. The regex is a Java string, so each regex backslash requires two backslashes in the source.

STYLE-RUN SELECTOR · STYLE REQUESTS ONLY

Match text runs by their current appearance

TextStyleRequest.runsWhere()
        .whereTextContains("Total")
        .whereFont("Helvetica-Bold")
        .whereSize(12.0, 0.25)
        .whereFillColor(PdfColorRequest.gray(0.0));

Selection: runs containing “Total” in Helvetica Bold, black, at 12 points within a tolerance of 0.25 points. Style-run selection is available only on TextStyleRequest.

A style-run selector requires at least one where… filter. Inference: the request model stores all supplied fields in one filter object, which suggests that a run must satisfy every supplied field. Verify the combination behavior with the target PDF.

PAGE SCOPE

Restrict eligibility by page

TextReplaceRequest.builder()
        .pages(2)
        .literal("Draft");

Selection: “Draft” only on page 2. Calling the completed operation through pdf.page(2).text() is the page-scoped alternative.

03 · Layout

Control the layout after an edit

The layout mode controls whether reflow is attempted and whether fallback is allowed. When the mode uses reflow, the reflow profile controls how the selected text unit is composed. These are separate decisions.

sourceAnchored

Keep every unaffected glyph at its original coordinates. Replacement text starts at the selected source anchor. Following text does not move, so a longer replacement can overlap it.

reflowWhenSupported

Attempts reflow. If reflow is unavailable for the selected unit, PDFDancer applies the edit source-anchored and returns a warning that identifies the fallback.

requireReflow

Requires reflow. If reflow cannot be applied to a selected unit, PDFDancer reports a text-reflow error for that unit and does not apply a source-anchored fallback.

SOURCE-ANCHORED · FIXED SOURCE GEOMETRY

Keep unaffected text at its original coordinates

TextEditResponse response = pdf.text().replace(
        TextReplaceRequest.literal("Q1", "First quarter")
                .sourceAnchored()
                .build());

Revenue in Q1 increased 18%.

The selected token occupies a small source range.

→
After · same anchor

The replacement expands, but “increased 18%” stays fixed and is overpainted.

Recommendation: use this mode for same-length or shorter substitutions, or for isolated text with enough empty space after it. A longer inline replacement can overlap unchanged text.

REFLOW · UNIT-NATIVE COMPOSITION

Recompose the edited text unit

TextEditResponse response = pdf.text().replace(
        TextReplaceRequest.literal("Q1", "First quarter")
                .reflowWhenSupported(TextLayoutRequest.Profile.BODY_TEXT)
                .hyphenationEnabled(false)
                .build());
Before

Revenue in Q1 increased 18% as demand improved across every region.

Original paragraph composition.

→
After · recomposed

Revenue in First quarter increased 18% as demand improved across every region.

Line breaks and spacing are recalculated inside the owning unit.

Recommendation: use reflow for prose replacements, insertions, or deletions when the paragraph may need new line breaks after the edit.

reflowWhenSupported · requireReflow

Set the outcome when reflow is unavailable

TextEditResponse response = pdf.text().delete(
        TextDeleteRequest.literal("obsolete paragraph")
                .requireReflow(TextLayoutRequest.Profile.DEFAULT)
                .build());

reflowWhenSupported

If reflow is unavailable, the unit receives a source-anchored edit and the response contains a warning.

attempt reflow→fallback + warning

requireReflow

If reflow is unavailable, the requested change is not applied to that unit and the response contains an error.

attempt reflow→no fallback + error

Recommendation: choose reflowWhenSupported only if a source-anchored fallback is acceptable. Choose requireReflow if the operation must not fall back. For either mode, inspect appliedLayoutMode(), warnings, and errors.

The three reflow profiles

The mode determines whether fallback is allowed. The profile determines which composition strategy PDFDancer uses when the request selects a reflow mode.

DETECTED LAYOUT PROFILE

DEFAULT

Chooses a profile from the detected layout model. Paragraphs and lists use composition suited to their alignment; protected or fixed-layout units use the no-reflow profile.

paragraph→body composition
list→list composition
heading / table→no reflow

Protected units include headings, labels, tables, tables of contents, callouts, structured or preformatted text, headers, footers, footnotes, page numbers, watermarks, and unknown layouts.

Recommendation: choose DEFAULT when the selection may contain different layout types and PDFDancer should choose a profile for each detected unit.

PARAGRAPH COMPOSITION

BODY_TEXT

Uses the body-text composer for every selected unit. It recomputes line breaks and can adjust tracking and word spacing within the profile’s constraints while preserving a readable paragraph measure.

A longer replacement is composed across the available paragraph width instead of remaining on one fixed line.

Recommendation: choose BODY_TEXT for known prose that should be recomposed as a paragraph. Do not force it onto headings, table cells, code, or preformatted text unless paragraph composition is the required result.

FIXED LINE BOUNDARIES

NO_REFLOW

Recomposes each existing line horizontally but preserves the original line boundaries. Following text moves to make room for a longer replacement; it never wraps onto a new line, and the line may extend beyond its original width.

Revenue in First quarter increased 18%.

Recommendation: choose NO_REFLOW when following text should move horizontally but every existing line break must remain unchanged. Inspect the right edge for overflow.

Result of replacing Q1 with First quarter

sourceAnchored(): following text stays fixed, so the replacement can overlap it. NO_REFLOW: following text moves right, but remains on the same original line. BODY_TEXT: following text moves and may wrap onto newly calculated lines.

Hyphenation is a per-operation override

Omit hyphenationEnabled(...) to inherit the selected profile’s setting. Explicit true or false changes only dictionary-generated discretionary hyphenation for this edit; it does not alter lexical hyphens supplied in the text. The override is invalid with sourceAnchored().

04 · Replace

Replace selected text

TextReplaceRequest requires exactly one replacement value: text, or image data with an affine transformation. A text replacement inherits the source appearance unless the request supplies appearance overrides.

TEXT → TEXT

Replace a literal everywhere in scope

TextEditResponse response = pdf.text().replace(
        TextReplaceRequest.literal("Acme Ltd", "Northwind GmbH")
                .reflowWhenSupported(TextLayoutRequest.Profile.DEFAULT)
                .build());

Result: every selected “Acme Ltd” occurrence becomes “Northwind GmbH”. The response reports the match count and applied-change count.

TEXT → IMAGE

Replace a placeholder with an image

PdfAffineTransform placement = PdfAffineTransform.builder()
        .scale(90, 32)
        .translate(3, -2)
        .build();

TextEditResponse response = pdf.text().replace(
        TextReplaceRequest.builder()
                .literal("{{logo}}")
                .replaceWithImage(new File("logo.png"), placement)
                .sourceAnchored()
                .build());

Result: each “{{logo}}” match is replaced by a 90 × 32 PDF-unit image. The transform offsets the image by (3, −2) from the visually leftmost boundary caret of the matched range.

Image replacement constraints

Image replacement supports only source-anchored layout and cannot be combined with text style overrides. The transform builder applies operations to points in invocation order and also supports rotateDegrees(...) and shear(...).

05 · Insert

Insert text at an anchor or coordinates

Anchor insertion places text at the caret before or after a match and inherits the appearance at that anchor. Coordinate insertion places text at PDF coordinates and requires a font and positive font size.

Anchor target

Select literal or regex text, then choose the caret immediately before or after the matched range.

Coordinate target

Supply a one-based page number and finite PDF coordinates. Rotation is optional. For a page-scoped call through pdf.page(n).text(), use coordinate(x, y) instead.

ANCHOR TARGET CONSTRUCTION

Choose the caret before or after a match

TextInsertRequest.after("Invoice total", " (including tax)");
TextInsertRequest.before("Invoice total", "Estimated ");
TextInsertRequest.afterRegex("\\bTOTAL\\b", " due");
TextInsertRequest.beforeRegex("\\bTOTAL\\b", "GRAND ");

Target: the caret immediately before or after each literal or regular-expression match. Each call returns a builder; add a layout mode and call build() before execution.

ANCHOR-BASED INSERTION

Insert after an anchor and reflow the paragraph

TextEditResponse response = pdf.text().insert(
        TextInsertRequest.after("Invoice total", " (including tax)")
                .reflowWhenSupported(TextLayoutRequest.Profile.BODY_TEXT)
                .build());

Result: “ (including tax)” is inserted immediately after every selected “Invoice total” range. The inserted text inherits the appearance at the anchor.

COORDINATE INSERTION

Place new text without an anchor

TextEditResponse response = pdf.text().insert(
        TextInsertRequest.at(1, 72, 720, "Reviewed")
                .rotationDegrees(0)
                .font("Helvetica-Bold")
                .size(12)
                .build());

Result: “Reviewed” is added to page 1 at PDF coordinates (72, 720) in 12-point Helvetica Bold. Coordinate insertion requires font(...) and size(...) because it has no inherited appearance.

06 · Delete

Delete selected text

Deletion removes the selected characters. With source-anchored layout, unaffected text remains fixed. With reflow, the selected text unit can be recomposed to close the gap.

REFLOWING DELETION

Delete a known phrase

TextEditResponse response = pdf.text().delete(
        TextDeleteRequest.literal("CONFIDENTIAL DRAFT")
                .reflowWhenSupported(TextLayoutRequest.Profile.DEFAULT)
                .build());

Result: each selected phrase is removed. The DEFAULT profile chooses a composition profile for each detected layout unit.

Replacement with an empty string also removes text

TextReplaceRequest.literal(source, "") is valid, but TextDeleteRequest communicates deletion more precisely and avoids implying that replacement styling might matter.

07 · Style

Set font, color, and spacing

Text styling uses six appearance fields: font, size, fill color, stroke color, character spacing, and word spacing. Styling, text replacement, and text insertion use the same field types but obtain their initial appearance differently.

COMPLETE APPEARANCE PATCH

Set every appearance field

TextEditResponse response = pdf.text().style(
        TextStyleRequest.literal("Payment overdue")
                .font("Helvetica-BoldOblique")
                .size(13)
                .fillColor(PdfColorRequest.rgb(0.75, 0.05, 0.05).alpha(0.95))
                .strokeColor(PdfColorRequest.gray(0.0))
                .characterSpacing(0.4)
                .wordSpacing(1.0)
                .sourceAnchored()
                .build());

Result: every selected phrase receives the specified font, size, fill color, stroke color, character spacing, and word spacing. Bold and italic are part of the font selection; the request has no separate bold or italic fields.

REPLACEMENT AND INSERTION OVERRIDES

Override inherited appearance

// Replacement: inherit the source style, then override two fields.
TextReplaceRequest.literal("{{status}}", "APPROVED")
        .font("Helvetica-Bold")
        .fillColor(PdfColorRequest.rgb(0.05, 0.45, 0.25));

// Anchor insertion: inherit the anchor style, then override one field.
TextInsertRequest.after("Balance due", " — overdue")
        .fillColor(PdfColorRequest.rgb(0.75, 0.05, 0.05));

Builder state: both calls return incomplete request builders. Add layout configuration as needed, then call build() and execute the request.

Color spaces

PdfColorRequest.rgb(0.2, 0.4, 0.8)
PdfColorRequest.cmyk(0.7, 0.2, 0.0, 0.1)
PdfColorRequest.gray(0.35)
PdfColorRequest.rgb(1, 0, 0).alpha(0.6)

Every component and optional alpha value must be finite and between 0.0 and 1.0.

Spacing reset

TextEditResponse response = pdf.text().style(
        TextStyleRequest.literal("Normal text")
                .resetSpacingOverrides(true)
                .build());

resetSpacingOverrides(true) cannot be combined with characterSpacing(...) or wordSpacing(...) in the same style request. Replacement uses the equivalent resetSpacingOverrides() builder method.

CUSTOM FONT

Register a TTF, then use its returned name

String fontName = pdf.registerFont(new File("SourceSans3-Regular.ttf"));

TextEditResponse response = pdf.text().style(
        TextStyleRequest.literal("Quarterly report")
                .font(fontName)
                .size(14)
                .build());

Result: the TTF is registered in the active session, and the selected text is changed to the registered font at 14 points.

Font name is not a guarantee of glyph coverage

A font can be registered successfully yet lack a glyph required by replacement text. Treat a response with matches but zero changes and a font-related error as a failed edit, not a successful no-op.

08 · Verification

Check the response before saving

TextEditResponse reports how many targets matched, how many changes were applied, which pages changed, and any available per-change diagnostics.

AccessorPrecise meaningWhat to check
matched()Number of selector matches or insertion targets reported by the operation.Zero usually means the selector or scope found nothing.
changed()Number of reported changes actually applied.Do not assume it equals matched().
pagesChanged()One-based page numbers reported as changed.Confirm scope, especially for document-wide requests.
change()Per-change diagnostics: page, operation, source/result text, requested and applied layout, element IDs, generated element IDs, and reflow-unit IDs.Use appliedLayoutMode() to verify layout behavior.
warnings()Nonfatal operation diagnostics with page, code, message, element IDs, and reflow-unit IDs.Review before accepting visual fidelity.
errors()Failure diagnostics with the same location/context fields.Any error means at least part of the requested result was not applied.

MINIMAL VERIFICATION

Reject incomplete results

TextEditResponse response = pdf.text().replace(
        TextReplaceRequest.literal("Draft", "Final")
                .build());

if (response.matched() == 0) {
    throw new IllegalStateException("Source text was not found");
}

if (response.errors() != null && !response.errors().isEmpty()) {
    for (TextOperationDiagnostic error : response.errors()) {
        System.err.printf("%s on page %s: %s%n",
                error.code(), error.page(), error.message());
    }
    throw new IllegalStateException("Text edit was not fully applied");
}

if (!response.matched().equals(response.changed())) {
    throw new IllegalStateException("Not every match changed");
}

Behavior: execution continues only when the operation reports at least one match, no errors, and the same number of matches and applied changes.

LAYOUT VERIFICATION

Inspect requested versus applied layout

if (response.change() != null) {
    for (TextEditChangeDiagnostic change : response.change()) {
        System.out.printf(
                "page=%d requested=%s profile=%s applied=%s hyphenation=%s%n",
                change.page(),
                change.requestedLayoutMode(),
                change.requestedLayoutProfile(),
                change.appliedLayoutMode(),
                change.effectiveHyphenationEnabled());
    }
}

Output: each available change diagnostic prints the requested mode and profile, the applied mode, and the effective hyphenation setting.

A no-match result is not necessarily an API failure

matched() == 0 and changed() == 0 can be a valid operation result. Whether it is acceptable is an application-level decision. In a required document transformation, treat it as a failed precondition.

09 · Save

Save the edited PDF

Every applied edit changes the active session. After checking the operation responses, write the current session state to a file or retrieve it as bytes.

SAVE TO DISK

Write the current session state to disk

pdf.save("output.pdf");

Result: output.pdf contains every change applied to the active session before this call.

In-memory alternative

Use byte[] editedPdf = pdf.getFileBytes(); when the PDF will be returned from an HTTP endpoint, written by another storage API, or passed to a downstream process.

10 · Pitfalls

Account for PDF text structure

PDF text is positioned as glyphs and text runs. Selection, font encoding, and layout therefore need explicit verification.

Visible text may be fragmented

General PDF-format inference: a phrase that looks continuous can be stored as multiple text runs or glyph sequences. If a literal selector returns no matches, inspect the extracted structure and test a narrower literal or regular expression. The response is the authoritative match result.

Scope can be broader than intended

pdf.text() is document-scoped. Prefer pdf.page(n).text(), pages(...), and wholeWords(true) when the target must be constrained.

Long source-anchored text can collide

Source-anchored edits leave unaffected glyphs at their original coordinates. A longer replacement can overlap following text. Choose a reflow mode when following text must move, and inspect the output.

Verify the applied layout mode

For reflowWhenSupported, inspect appliedLayoutMode(). If the result requires reflow, request requireReflow and reject returned errors or an unexpected applied mode.

Fonts must encode the result

The requested font must support every replacement glyph. A response can report a match but no change when the replacement cannot be encoded.

Image replacement is not text styling

It accepts image data and a PDF affine transform, supports source-anchored layout only, and rejects replacement style fields.

Regexes can match too much

Remember Java string escaping, use boundaries where appropriate, and compare the returned match count with the expected count before saving.

No match, warning, and error differ

No match means nothing was selected. A warning describes a nonfatal condition. An error reports that requested work was not applied. Handle each state separately.
